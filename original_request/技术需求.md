# 学院学生事务一站式小程序  
**需求 & 大致解决方案 v4.0（业务导向版）**  
**日期**：2026年3月22日  
**目标**：让学生少跑腿、少问人，老师少重复解释  
**用户**：约1000名学生 + 团委/班主任/助教等少量管理员

## 1. 总体原则（所有方案必须遵守）

- 学生敏感信息（姓名、学号、身份证相关、入党/入团时间等）必须存储在学校自有数据库（人大金仓），小程序侧只存映射关系（如微信标识与学号的对应）。
- 消息通知统一使用微信小程序的订阅消息能力。
- 尽量避免依赖外部大模型做核心业务逻辑（政策解释、证明生成等），优先用结构化数据 + 标准模板。
- 不做任何自动抓取外部公众号/网站的动作。
- 不做成绩单解析、培养方案自动比对、选课推荐等高复杂度功能。

## 2. 核心模块 & 大致解决方案方向

### 模块一：政策咨询与知识库

**需求**：学生输入问题或关键词 → 快速得到标准答案、官方文件链接或办理指引  
**解决方案方向**：
- 建立结构化的知识库（管理员维护常见问题 + 标准回复 + 附件链接）
- 支持关键词搜索或模糊匹配返回结果
- 搜索不到时引导联系老师或跳转咨询入口
- 选择方式：
  - 纯数据库 + 简单匹配（最稳、最低成本）
  - 前端轻量模糊搜索库（离线快）
  - 如需更智能：接入可控的小模型 + 严格检索优先（可选，后期加）
**一个简单的实现方法，对输入数据进行简单分段和分词，然后使用分词搜索，使用elasticsearch或postgre（技术栈更轻）**

### 模块二：党团事务进度查询与提醒

**需求**：学生查看自己当前所处阶段 + 系统自动提醒关键节点（如思想汇报提交时间）  
**解决方案方向**：
- 维护学生党团阶段信息（管理员批量导入或逐个更新）
- 学生端展示清晰的流程进度（类似时间轴或步骤条）
- 根据时间规则自动触发提醒（例如积极分子满一定月数等）
- 通知方式：小程序订阅消息（需用户提前订阅对应模板）
- 选择方式：
  - 云函数/服务器定时任务扫描触发（主流）
  - 数据库定时器或触发器（如果数据库支持）
**当前收敛原则**：
- v0 先做“入团 + 入党”两条稳定主线，共用一套阶段流转模型
- 理论自测暂不纳入当前阶段
- 提醒能力只覆盖明确规则的关键节点，不做复杂个性化编排
**需要进一步确认这里的状态流程（状态机）**

### 模块三：电子证明生成

**需求**：学生选择证明类型 → 系统自动填充个人信息 → 生成可下载的PDF文件  
**解决方案方向**：
- 从学校数据库拉取学生基础信息（需授权校验）
- 使用标准模板自动填充后生成PDF
- 学生预览后保存到手机或小程序文件
- 选择方式：
  - 小程序端直接生成（轻量库实现）
  - 服务器/云函数端生成（排版更稳定，适合复杂证明）
**当前收敛原则**：
- v0 仅支持少量标准证明模板，不做开放式模板设计器
- 证明内容优先使用结构化字段填充，不依赖大模型生成正文
- 生成结果以 PDF 下载/预览为主，后续如有需要再接入审批联动

### 模块四：信息发布与精准推送

**需求**：管理员发布学校/学院通知 → 根据学生属性（如年级、专业、毕业年份）精准推送  
**解决方案方向**：
- 管理员后台录入通知内容 + 选择推送范围（标签或条件组合）
- 系统根据学生信息筛选目标群体后推送
- 推送通道：小程序订阅消息
- 选择方式：
  - 手动录入 + 标签系统（最简单可靠）
  - 结合学生基础属性做条件筛选（年级、专业等）
**当前收敛原则**：
- 不做公众号或网站自动抓取
- 支持发布公共消息，并允许管理员手动附带公众号文章链接、学校官网链接或其他官方外链
- 公共消息与精准投放共用同一发布模型，公共消息可视为“面向全体学生”的特例

### 模块五：简单审批流程

**需求**：学生提交请假、盖章等申请 → 相关老师在线审批 → 状态实时更新  
**解决方案方向**：
- 学生提交申请 → 流转到当前审批人
- 审批人小程序内查看、通过/驳回
- 支持超时提醒或自动转交（规则需明确）
- 选择方式：
  - 简单状态流转（适合线性审批）
  - 轻量工作流配置（如果未来流程变复杂，可扩展）

## 3. 明确不做的部分（避免后期争议和难以实现的需求）

- 任何形式的自动抓取公众号/网站内容（相关的获取信息的方式，可以提供一个页面放上链接，主要是微信不允许抓取公众号内容）
- 成绩单识别、培养方案比对、选课推荐（应当由教务平台负责，而且手动实现更符合个人需求）
- 邮件/短信批量通知（成本 & 合规问题）
- 依赖通用大模型做政策解读或生成内容
- 党团模块理论自测（当前阶段先不做）

## 4. 系统整体选择方向（供决策参考）

**后端 & 数据库组合**：
- 微信云开发为主 + 学校金仓做核心数据（开发快、通知稳定、敏感数据可控）
- 自建后端服务 + 金仓全托管（数据最自主，但运维成本高）
- 纯云开发（仅用于原型或非敏感场景，不推荐正式版）

**身份绑定方式**：
- 学生手动输入学号 + 姓名 + 简单校验
- 结合学校已有系统验证码（如果有接口）
- 暂不考虑微信直接返回实名信息（平台不支持）

**PDF生成位置**：
- 小程序前端生成（响应快、轻量）
- 云函数/服务器生成（排版控制更好）

**实现方式**：
- 明确业务的一些具体细节，比如提交的文件格式、提醒的规则
- 在明确后端所需实现的api接口后再进行开发，api一旦定下，除非需求更改的情况下，不作变化
- 各个模块耦合性很低，确定好最基础的用户数据类型后，约定好技术栈即可分别开发后端相关模块
- 在开发完后端各模块，并测试通过之后，使用**框架**开发微信小程序，避免原生开发
